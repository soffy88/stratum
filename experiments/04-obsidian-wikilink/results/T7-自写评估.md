# T7: 自写 Obsidian 插件可行性评估

**评估者**: Claude Code (基于 Obsidian Plugin API 官方文档 + 训练数据)
**评估时间**: 2026-05-16
**评估对象**: STRATUM_SPEC §4.3 扩展 wikilink 语法的自写插件实现可行性

---

## 1. API 支持度

Obsidian Plugin API **不提供直接拦截 wikilink 解析器的钩子**。内置的 wikilink 解析是 Obsidian 核心的私有逻辑，插件无法在渲染链最上游替换解析规则。

可用的 workaround 有两条路径：

**路径 A: `registerMarkdownPostProcessor()`**
在 Markdown 渲染为 HTML 之后介入，用 `querySelectorAll('a.internal-link')` 找到所有内部链接，检查其 `href` 属性，如果是形如 `concept/01HY.../xiang-yu` 的扩展格式，则拦截 click 事件并调用 `workspace.openLinkText()` 跳转。这个 API 是**公开稳定接口**，适用于 Reading View。

**路径 B: `registerEditorExtension()` + CodeMirror 6 扩展**
在 Live Preview (编辑器模式) 下，需要通过 CodeMirror 6 的 `ViewPlugin` / `Decoration` 机制，在编辑器层面渲染自定义链接样式。实现复杂度显著更高，但体验更完整。

## 2. 解析 ULID 路径的复杂度

给定链接文本 `[[concept/01HY0000000000000000000001/xiang-yu|项羽]]`，解析逻辑如下：

```typescript
// 伪代码: 从 wikilink href 提取结构
function parseStratumLink(href: string): StratumRef | null {
  // href 经 Obsidian 处理后形如: "concept/01HY.../xiang-yu"
  // 带段落锚点时形如: "substrate/01HY.../shiji-007#A1B2C3"
  const match = href.match(
    /^(substrate|concept|note)\/([0-9A-HJKMNP-TV-Z]{26})\/([a-z0-9-]+)(?:#([A-Z0-9]+))?$/
  );
  if (!match) return null;
  return {
    layer: match[1] as Layer,
    ulid: match[2],
    slug: match[3],
    paragraphId: match[4] ?? null,
  };
}
```

**性能估算**: 正则匹配单次 < 0.01ms，1000 次调用约 5-10ms，不可感知。瓶颈不在解析，而在后续的 ULID→文件路径查找（见第 3 节）。

## 3. 跳转目标定位

**方案 A: 每次跳转时全库扫描 frontmatter**

- 每次点击链接时，遍历 `app.vault.getMarkdownFiles()`，用 `app.metadataCache.getFileCache(file)?.frontmatter?.id` 比对 ULID。
- 全库 1000 个文件时，单次扫描约 20-50ms（同步），用户感知跳转延迟。
- **不推荐**: 库规模增大后体验变差，且重复扫描浪费。

**方案 B: 构建 ID→path 索引 + 缓存 (推荐)**

```typescript
class UlidIndex {
  private map: Map<string, string> = new Map(); // ulid → vault-relative path

  async build(app: App) {
    for (const file of app.vault.getMarkdownFiles()) {
      const meta = app.metadataCache.getFileCache(file);
      const id = meta?.frontmatter?.id;
      if (id) this.map.set(id, file.path);
    }
  }

  resolve(ulid: string): string | undefined {
    return this.map.get(ulid);
  }
}
```

**文件改名/移动的更新机制**:

```typescript
// 在 onload() 中注册
this.app.vault.on('rename', (file, oldPath) => {
  const meta = this.app.metadataCache.getFileCache(file);
  const id = meta?.frontmatter?.id;
  if (id) this.index.map.set(id, file.path); // 路径更新, ULID 不变
});
```

由于 ULID 存在 frontmatter 且文件改名不改 frontmatter，rename 事件可以准确更新 path，索引始终有效。这是 STRATUM "ULID 不可变，slug 可变"设计的核心优势。

## 4. graph view 集成 (关键判断)

Obsidian graph view 的数据来源是 `app.metadataCache.resolvedLinks`，这是一个 `Record<string, Record<string, number>>` 形式的内部属性（文件→引用文件→引用次数）。

**能否注入**: **技术上可以，但依赖内部 API**。

```typescript
// 非公开 API，可能随版本变化
const resolved = (app.metadataCache as any).resolvedLinks;
if (resolved[sourceFilePath]) {
  resolved[sourceFilePath][targetFilePath] = 1;
}
```

社区已有先例（如 `Strange New Worlds` 插件）用类似方式让 graph 显示自定义关联线。方法可行，但 Obsidian 官方不保证 `resolvedLinks` 的结构稳定性。

**结论**: graph view 集成**不是 hard limitation**，是 **soft limitation（依赖未公开 API）**。MVP 阶段可跳过，完整版需接受维护风险。

## 5. 工作量估算

| 范围 | 功能 | 工作量 | 维护风险 |
|------|------|--------|---------|
| MVP (跳转) | Reading View 链接拦截 + ULID 索引 + `openLinkText` 跳转 + 段落锚点 `#` 支持 | **3-5 天** | 低 (使用公开 API) |
| 完整 (含 graph) | MVP + Live Preview 渲染 + graph view 注入 + hover preview + auto-complete | **15-20 天** | 中 (Live Preview 和 graph 依赖内部 API) |

**MVP 3-5 天的分解**:
- Day 1: 项目脚手架 + ULID 索引构建 + rename 事件监听
- Day 2: MarkdownPostProcessor 注册 + 链接拦截 + 跳转逻辑
- Day 3: 段落锚点 (#A1B2C3) 跳转支持
- Day 4-5: 测试 + 边界情况处理 + README

**长期维护风险因素**:
1. Obsidian 每月发大版本，core API 变更概率约 10-20%/年（低风险部分）
2. `resolvedLinks` 内部结构每隔 1-2 个大版本可能调整（中风险，仅影响 graph 功能）
3. 插件维护需跟进 API 变更，估计每年 1-2 次小修

---

## 结论

**推荐: 自写 MVP 版本 (3-5 天)**，完整版推迟到批 4。

**理由**:
1. MVP 所需的所有 API (`registerMarkdownPostProcessor`, `vault.on('rename')`, `workspace.openLinkText`) 都是 Obsidian **公开稳定接口**，维护风险低。
2. STRATUM §4.3 的核心价值在于 "ULID 不变，引用不断链"——MVP 就能实现这个价值。
3. graph view 集成（依赖内部 API）对日常使用不是阻塞项，可在批 4 接口层一起做。

**是否阻塞批 3-4 启动**: **不阻塞**。批 3 是流水线 MVP，不依赖 Obsidian 插件。批 4 接口层才需要插件，届时 MVP 插件应已完成。

**重要前提**: T1-T6 手动测试若发现某个现有插件（如 Various Complements + Dataview 组合）已能满足需求，则无需自写——本评估仅在 T1-T6 全部 FAIL/PARTIAL 的情况下适用。
