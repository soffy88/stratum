# ADR-021 — Stratum 跨机部署拓扑

**状态**: 已确认 (2026-05-20)
**决策者**: Wiki (Stratum chief advisor)
**影响范围**: Phase 11 外挂能力部署、ADR-019 修订、ADR-016 扩展

---

## 背景

### 设备拓扑校准 (2026-05-18)

Wiki 完成设备清点：

| 设备 | GPU | RAM | 主用途 |
|---|---|---|---|
| 笔记本 | 4G VRAM (轻 GPU) + 24G RAM | 24G | 日常开发, Stratum 主体 |
| 主机 | 10G VRAM + 32G RAM | 32G | 闲置, 重 GPU 任务候选 |
| Singapore VPS | CPU only | — | 代理, 公网备份 |

**触发原因**:

1. **ADR-019 失效**: ADR-019 基于 "同 Docker network" 假设设计外挂集成方案。笔记本 + 主机跨机场景下，同 Docker network 无法跨物理设备，ADR-019 核心通信假设不成立。
2. **Phase 11 外挂启动条件**: Phase 11 引入 `oprim.external.*` 能力 (whisper / TTS / SD / searxng / ollama)，这些外挂的 GPU 需求超出笔记本 4G VRAM，必须在 Phase 11 启动前锁定外挂部署位置。
3. **主机 GPU 资源利用**: 主机 10G VRAM 长期闲置，正好承载重 GPU 外挂，无需新增硬件成本。

---

## 决定

### 核心决定: 三层部署拓扑 + Tailscale mesh 跨机通信

#### 设备角色分层

| Layer | 设备 | 角色 | 承载组件 |
|---|---|---|---|
| **Layer A** | 笔记本 | Stratum 主体 + 轻外挂 | stratum-main, postgres, redis, rabbitmq, lancedb, searxng |
| **Layer B** | 主机 | 重外挂 (GPU 密集) | whisper-large, F5-TTS, SD-webui, ollama |
| **Layer C** | Singapore VPS | 代理 + 公网备份 | sing-box, 备份接收端 |

**故障域原则**: 各 Layer 独立。Layer A 是 Stratum 单点 (笔记本常开)，Layer B/C 允许离线，Stratum 主体降级运行。

#### 跨机通信方案

**Tailscale mesh + Magic DNS**

- 三台设备均加入同一 Tailscale tailnet
- Stratum 主体通过 Magic DNS 调用外挂：`stratum-host.tail-net.ts.net`
- 不依赖公网 IP，不穿越 NAT，延迟约 5–15ms (LAN 直连实测预期)
- 跨机容器间通信走 Tailscale 虚拟网卡，主机侧容器绑定 `0.0.0.0:PORT`

## 外挂分配决策

基于主机 10G VRAM 实测能力：

| 外挂 | 估算 VRAM | 分配 Layer | 备注 |
|---|---|---|---|
| whisper.cpp small/medium | ~2G | Layer B (主机) | 笔记本 4G VRAM 可跑，但归主机统一管理；笔记本作冷 fallback |
| whisper.cpp large-v3 (Q4/Q5 量化) | ~4–5G | Layer B (主机) | 使用 `ggml-large-v3-q5_0` 等量化模型；全精度约 10G (紧贴极限)，量化后精度损失极小且 whisper.cpp 默认部署方式 |
| F5-TTS | ~6–8G | Layer B (主机) | 推荐 TTS 方案；实测约 6.4G (800 字段落，16G 卡)，10G 卡单独运行可行 |
| fish-speech | ~6–12G (版本依赖) | **不部署** | v1.5.0 仅 1.8G，v1.5.1+ 约 5.8G+，S2 模型官方推荐 24G；版本差异大，风险高；用 F5-TTS 替代 |
| Stable Diffusion 1.5/2.1 | ~4–6G | Layer B (主机) | 主力图像生成 |
| SDXL | ~12G | **不部署** | 超出 10G VRAM；需要时用云 API 或 SD 1.5 替代 |
| searxng | CPU only | Layer A (笔记本) | 轻量，与主体同机更低延迟 |
| ollama qwen3:14b-q4 | ~8.3–10G | Layer B (主机) | Q4_K_M 权重 ~8.3G + KV cache；10G 卡上下文受限约 4K token，满足 Stratum 使用场景 |
| hevi | 待定 | 不归 Stratum 管 | hevi 经理人独立决定部署位置 |

**GPU 并发约束**: whisper large-v3 全精度约 10G (不推荐)；量化版 ~4–5G，与 F5-TTS (~6–8G) 不可并发但与 SD/ollama 有机会共存 (需实测)。调度策略见未决问题 Q15。

## 理由

### 选 Tailscale mesh，不选其他方案

| 方案 | 优点 | 缺点 | 决定 |
|---|---|---|---|
| Tailscale mesh + Magic DNS | 零配置 NAT 穿透，Magic DNS 稳定，Wiki 已有账号 | 依赖 Tailscale 控制面 (可接受，离线时 LAN 直连仍通) | ✅ 选用 |
| Docker overlay network (Swarm) | 原生 Docker | 需要 Swarm 初始化，运维复杂度高 | ❌ |
| ZeroTier | 功能相似 | 延迟略高，控制面不如 Tailscale | ❌ |
| 直接公网 IP + 防火墙 | 无依赖 | 暴露端口，安全风险，Singapore VPS 绕行延迟高 | ❌ |
| wireguard 手动配置 | 无控制面依赖 | 手动密钥管理，Magic DNS 需自建 | 备选，Tailscale 不可用时回退 |

### Layer A 用 `network_mode: host`

笔记本上 Stratum 主体容器使用 `network_mode: host`，直接共享宿主机 Tailscale 虚拟网卡。避免 Docker bridge 额外 NAT，简化容器内对外访问。

### searxng 放 Layer A

searxng CPU only，与 Stratum 主体同机延迟更低 (同机内网 vs Tailscale 5–15ms)。不占 Layer B GPU。Layer B 专注 GPU 重负载。

## ADR-019 修订

ADR-019 (外挂能力架构，同 Docker network 方案) 以下条款被本 ADR 修订：

### 保留条款 (ADR-019 原文有效)

- **解耦原则**: 各外挂独立 container，独立故障域
- **各经理人独立**: Stratum advisor 只规范接入接口，不管外挂内部实现
- **本地优先**: 不依赖云 API (searxng/whisper/TTS/SD 全本地)
- **MCP 协议优先通信**: 外挂经理人通过 MCP 向 Stratum 注册工具

### 变更条款

| ADR-019 原条款 | ADR-021 修订 |
|---|---|
| 外挂与 Stratum 主体同 Docker network 部署 | **废止**: 改为 Tailscale mesh 跨机通信 |
| "同机 docker network 内网延迟 < 5ms" | **废止**: 改为 "Tailscale LAN 延迟 5–15ms (跨机)" |
| 外挂 docker-compose 并入 stratum 主 compose 文件 | **废止**: Layer B 外挂独立 `docker-compose.layer-b.yml`，不并入 Layer A |

### 新增条款

- **跨 Layer 故障降级**: 见本 ADR §故障降级策略
- **外挂 base_url 通过环境变量注入**: `STRATUM_WHISPER_HOST` / `STRATUM_TTS_HOST` / `STRATUM_SD_HOST` / `STRATUM_OLLAMA_HOST`，不硬编码 docker network 内网地址

## 数据流向与隐私

与 STRATUM_SPEC §13.4 保持一致：

- **substrate 数据始终在 Layer A + GDrive** (Wiki 个人网盘)，不流入 Layer B/C
- **外挂调用只传输任务数据**: transcribe 只传输 audio 片段，TTS 只传输文本，SD 只传输 prompt；不传 substrate 完整内容
- **外挂内不持久化任务数据**: 任务完成后清理临时文件
- **Tailscale mesh 不出 Wiki 个人网络**: 三台设备均在同一 tailnet，数据流不经过公网第三方节点 (Tailscale relay 仅在 NAT 不通时中转，LAN 直连时点对点加密)

### Magic DNS 主机名约定

| 设备 | Tailscale Magic DNS |
|---|---|
| 笔记本 (Layer A) | `stratum-main.tail-net.ts.net` |
| 主机 (Layer B) | `stratum-host.tail-net.ts.net` |
| Singapore VPS (Layer C) | `stratum-vps.tail-net.ts.net` |

> **注**: 实际 tailnet 域名在 Tailscale Admin Console 的 DNS 页确认。上述为示意，部署时替换。
>
> **Docker Magic DNS 注意事项** (Wave 2 调研发现，Tailscale GitHub #14467/#10216)：Docker 容器默认不继承宿主机的 Tailscale Magic DNS。Layer A 使用 `network_mode: host` 可绕过此问题 (共享宿主机网络栈)。Layer B 容器绑定 `0.0.0.0:PORT` 仅需宿主机 Tailscale IP 可达，不需容器内 Magic DNS。若未来 Stratum 主体容器改为 bridge 网络，需在容器 `/etc/resolv.conf` 加入 `100.100.100.100` 或使用 Tailscale sidecar 模式 (`TS_ACCEPT_DNS=true`)。
>
> **WSL2 内核前向风险**: Linux kernel 6.8+ 部分版本 Tailscale kernel-mode 网络有已知问题，需 `TS_USERSPACE=true` 回退。当前 WSL2 kernel 6.6.114 未受影响，但升级内核时需确认。

## docker-compose 示例

### Layer A — 笔记本 (`docker-compose.layer-a.yml`)

```yaml
# ~/projects/stratum/docker-compose.layer-a.yml
# Stratum 主体 + 轻外挂 (searxng)
# network_mode: host 共享 Tailscale 网卡，直接访问 stratum-host.ts.net

services:
  stratum-main:
    image: stratum/stratum-main:v2.5.0
    network_mode: host
    environment:
      STRATUM_WHISPER_HOST: "http://stratum-host.tail-net.ts.net:9000"
      STRATUM_TTS_HOST: "http://stratum-host.tail-net.ts.net:9001"
      STRATUM_SD_HOST: "http://stratum-host.tail-net.ts.net:9002"
      STRATUM_OLLAMA_HOST: "http://stratum-host.tail-net.ts.net:11434"
    volumes:
      - ~/.stratum:/home/app/.stratum

  postgres:
    image: postgres:16
    network_mode: host
    environment:
      POSTGRES_DB: stratum
      POSTGRES_USER: stratum
      POSTGRES_PASSWORD_FILE: /run/secrets/pg_password
    volumes:
      - stratum_pg_data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    network_mode: host

  rabbitmq:
    image: rabbitmq:3-management-alpine

  searxng:
    image: searxng/searxng:latest
    network_mode: host
    volumes:
      - ./config/searxng:/etc/searxng

volumes:
  stratum_pg_data:
```

## 故障降级策略

各 `oprim.external.*` 客户端实施 circuit_breaker (4O v0.3 §E.1 规范)，N 次连续失败后 fail-fast，不堵塞 Stratum 主体。

| 故障场景 | Stratum 行为 | 数据安全 |
|---|---|---|
| Layer B 离线 (主机关机/网络断) | Stratum 主体正常运行；whisper/TTS/SD/ollama 不可用；Agent 路由跳过外挂任务，排队等待恢复；circuit_breaker 开路后 fail-fast | Layer A + GDrive 完整，无数据丢失 |
| Layer B 单服务崩溃 (e.g. SD OOM) | 仅该服务不可用，其他外挂正常；Stratum 降级路由 | 同上 |
| Layer C 离线 (VPS) | 代理不可用；sing-box 相关请求失败；国内访问公网 API 受影响 | substrate 数据不在 VPS，无影响 |
| Layer A 离线 (笔记本) | Stratum 全停 (Layer A 是主单点) | 数据安全: substrate 在 GDrive；恢复时 `git pull` + 重启即可 |
| Tailscale 控制面不可达 | LAN 直连仍通 (Tailscale 点对点，控制面仅用于密钥协商)；Magic DNS 可能降级为 IP 直连 | 无数据影响 |
| Tailscale mesh 完全断开 | Layer B 不可达；外挂全部不可用；Stratum 降级运行 | 无数据影响 |
