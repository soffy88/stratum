# 第一部分 · 定位正名（被其它文档引用的权威表述）

## 1.1 本体是发现，不是规定（实证结论）

本体工程史上的路线之争与胜负：
- **工程派**（Musen 等）：本体 = 任意的"概念化规格"，各定各的 → **必然增殖**（千千万万互不相容的本体，谁的都不算数）——正是 Wiki 归谬指出的死局，也是语义网烂摊子的根源。微软 Ontology-Playground 让人手画本体，走的是这条路（量产玩具本体）。
- **实在论派**（Barry Smith / BFO / OBO Foundry）：**"本体应当描述实在，而不是语言、概念或心理表征"**；**"实在是评估和对齐本体的共同基准"**；**正交性原则：每个领域应当收敛到单一本体**。
- **胜负由实践判定**：实在论派的 OBO Foundry（基因本体 GO 等 100+ 互操作本体）是史上最成功的本体生态。

**→ 结论：真正的本体 = 领域的真实类型结构，是被发现的、以实在为基准的、一域收敛到一个的。**

## 1.2 本体 vs 本性：都是发现，深度不同

| | 本体 ontology | 本性 invariant |
|---|---|---|
| 发现什么 | **域内**的真实类型结构（有哪些概念/什么层级/什么判别维度/什么关系） | **跨域**的不变内核 |
| 基准 | 领域积累的真知识（文献/实在） | 三层互证（语义+拓扑+逻辑） |
| 收敛 | 一域一个（正交性） | ≥2 概念共有才升华 |

**两者不是"规定 vs 发现"，是发现的深度不同。** 旧表述"本体=人预先规定的框架"作废。

## 1.3 ★AII 与 OBO 生态的精确对位（AII 本来就是本体发现机）

| OBO 生态（最成功实践） | AII |
|---|---|
| **BFO**（上层本体：极简、领域中立的元语法） | **AII-KNOWLEDGE-ONTOLOGY-002 元本体**（KU六分类/概念/相/本性的定义）——**人定但不任意**：是对"知识的真实结构"的假设，**可被证据修正**（已发生：nature→invariant、本性概念独立表→字段） |
| **领域参考本体**（GO 等：从领域真知识发现、一域一个、收敛） | **B仓 canonical 概念层 + 判别维度 + 第①层有向关系**——从书中**发现**的领域本体（A→B 管线 = 本体发现管线；sources 溯源 = 以文献实在为基准） |
| **正交性原则**（收敛机制，防增殖） | **判同"真正同一才合一" + 越读越厚**——千万本书进来收敛到**一个**领域本体，这就是 AII 不产出千千万万本体的机制 |
| （OBO 到此为止——正交 = 各域刻意不重叠） | **★第③④层跨域关系 + 本性 = OBO 明确不做的那一层**：发现正交领域之间共享的不变内核 |

**AII 的独特位置 = 实在论本体范式（下三层）+ 往下再挖一层（跨域本性）。**

# 第二部分 · 反增殖原则（规则，全 AII 适用）

```
R1 一域一本体：每个 discipline 的概念层收敛到一个（判同执行），绝不并行维护
   同一领域的多套概念体系。
R2 外部本体不整体采纳：任何外部本体/分类法（os-taxonomy、微软式手画、RDF导入、
   行业标准）进 AII 一律按【外部 judgment】方式处理（同 C仓 §2.4）：
   标来源（judgment_source=external_ontology + 具体出处）、grade=unverified、
   ★当对照不当权威——AII 的领域本体以自己从书中发现的为准。
R3 RDF/OWL 只作对外简化导出格式（将来若需与外部工具交换），绝不作主存储
   ——RDF 二元三元组表达不了 n 元超边和 invariant，AII 内部结构远比它丰富。
R4 元本体（ONTOLOGY-002）受同样纪律：它是关于知识真实结构的假设，
   证据推翻时修正（有先例），不是不可动的教条。
```

# 第三部分 · ★外部本体交叉检验（可实施，os-taxonomy 首个实例）

## 3.1 机制定位

AII 从书中发现的领域本体（概念层+层级），可与**外部标准本体/分类法对照**——**差异不自动改，只产出复核提示**。这给概念层/第①层加一条**独立检验线**（与本性层三层互证同一精神：多个独立观察者的发现结果互相对照）。

**对照的认识论地位**：外部本体 = 另一个观察者对同一领域实在的发现结果。一致 → 双方都可能对（增信）；冲突 → **不预设谁对**（书的证据 vs 外部本体，由人裁决），但冲突点是最值得看的地方。

## 3.2 首个对照源：os-taxonomy

```
os-taxonomy：1,590 个 micro-topic 节点、3,221 条 DAG 边的数学/教育先修图
  （已在库，license 已分析：ODbL produced-work 豁免，只读对照 + 产出 diff 报告
   属 produced work，可用；不修改不再分发该分类法本身）
对照对象：B仓数学领域的 canonical 概念层 + 第①层有向关系
  （尤其 prerequisite 关系——os-taxonomy 正是先修图，同类关系可直接对）
★时机依赖：M0 canonical + 步骤4.5 有向关系在数学领域跑完之后
  （对照的是去重归一后的概念层，不是 A仓原始概念——A仓有重复，对不上）
```

## 3.3 实施：三步（对齐 → diff → 复核报告）

### 步骤1 · 概念对齐（★本身就是判同问题，用判同纪律）
```
把 os-taxonomy 的 topic 节点映射到 B仓概念：
  ① 精确名匹配（英文名归一后 exact match）→ 高置信映射
  ② embedding 粗筛（BGE-M3，同 discipline 内余弦≥阈值）→ 候选映射
  ③ ★候选映射人工确认（经理人批量审）——映射本身是跨体系判同，
     宁不映射不错映射（错映射会制造假冲突/假一致，污染整个对照）
  ④ 映射结果落表：external_concept_map
     (ext_source, ext_node_id, concept_id, match_method, confidence, confirmed)
预期：只有部分节点能映射上（两边粒度/覆盖不同）——正常，未映射的不硬凑
```

### 步骤2 · diff（四类差异，分级）
```
D1 层级/先修方向冲突（★最强复核信号）：
   os-taxonomy 说 A 先修 B，B仓有向关系说 B derives/prerequisite A
   → 必有一方错（或映射错）→ 高优先复核
D2 外部有、AII 无的关系：
   os-taxonomy 里 A→B 有先修边，B仓两概念都有但无边
   → ★可能是 AII 漏抽的完整性线索（readout 漏了）→ 中优先复核
D3 AII 有、外部无的关系：
   → 可能 AII 发现了外部没覆盖的结构，也可能附会 → 低优先抽查
D4 覆盖差异：外部有的概念 B仓完全没有
   → 可能书源未覆盖该主题（正常）/ 可能抽取漏了 → 记录，供选书参考
```

### 步骤3 · 复核报告（★不自动改，Wiki/经理人裁决）
```
产出：ontology_crosscheck_report_<domain>_<date>.md
  · D1-D4 分级列出，每条附：两边的原始依据
    （B仓侧：sources 回查到书的原文；外部侧：os-taxonomy 节点）
  · ★裁决原则：书的证据优先（AII 以文献实在为基准；外部本体也可能错），
    但 D1 冲突必须给出解释（为什么书对/为什么外部对/还是映射错了）才能关闭
  · 采纳外部意见修改 B仓 → 走正常修正流程（dry_run + 回归网 + commit），
    修改理由记录"经外部本体对照发现"
红线：★对照程序对 B仓【只读】；任何修改由人走正常流程，绝不自动改
```

## 3.4 与既有机制的关系
```
评估集（EVALSET A类）：防已知的错重犯（回归网）
可视化（VIZ）：       人眼发现未知的错
★本体交叉检验（本SPEC）：用独立观察者的发现结果对照，暴露双方视角差
三者互补，都服务于同一件事：不让"看起来对"冒充"真的对"。
```

# 第四部分 · 执行顺序与红线

## 顺序
```
现在：      第一/二部分（定位+规则）即刻生效——master 等文档引用本文，
            旧表述"本体=人定框架"作废
M0+4.5 后： 第三部分实施（数学域概念层就绪 → os-taxonomy 对照跑第一轮）
将来：      每个新领域本体成型后，可找该领域的外部标准分类法做同样对照
            （经济学/物理学的标准分类法，届时再选源）
```

## 红线
```
1. 外部本体一律 unverified 外部 judgment：当对照不当权威（R2）
2. 概念对齐用判同纪律：宁不映射不错映射（错映射污染整个对照）
3. 对照程序对 B仓只读；修改由人走正常流程（dry_run+回归网），绝不自动改
4. D1 方向冲突必须解释才能关闭（不许"算了先放着"——冲突点是最值得看的）
5. 裁决以书的文献证据为基准（AII 是实在论本体：实在>任何本体包括外部的）
6. RDF/OWL 不作主存储（R3）
7. 每批 commit 必 push
```
